# ⚙️ 실시간 웹소켓 인프라 및 대규모 데이터 최적화기 (Module 4)

> **핵심 요약:** Fan-out 구조 개선을 통한 DB 커넥션 고갈 및 락(Lock) 리스크 원천 차단, 동기/비동기 관심사 분리를 통한 메인 트랜잭션 보호, IN 절 활용 벌크(Bulk) 조회를 통한 $O(1)$ 인메모리 매핑 혁신, 화면 레이턴시 최대 57% 단축 및 프론트엔드 연동 결함 격리

---

## 1. 개요
모듈 4에서는 앞서 구축한 친구 및 메시징 도메인을 확장하여, 시스템 전반의 실시간 커뮤니케이션을 담당하는 **실시간 웹소켓(STOMP) 알림 인프라**와 **다중 채널 실시간 갱신 시스템**을 전담하여 구현했습니다. 특히 대규모 트래픽 환경에서 발생할 수 있는 데이터베이스 병목을 해결하기 위해 백엔드 아키텍처를 전면 리팩토링하였으며, 모듈 4에서도 **DBA 역할**을 연속 수행하여 팀 내 인프라 결함과 파편화 문제를 주도적으로 돌파했습니다.

---

### 👥 팀 구성 및 협업 환경
* **팀 구성**: **백엔드 5명 (JSON 응답 설계) / 프론트엔드 2명**
* **담당 역할**: 웹소켓(STOMP) 실시간 알림 인프라 구축, 알림 라이프사이클 최적화, 대규모 조회 성능 개선 및 프로젝트 DBA 역할 리드
* **핵심 성과**: 단순 기능 구현을 넘어, 웹소켓 실시간 화면 갱신 시 발생하는 Fan-out 문제를 인메모리 벌크 아키텍처로 개선하여 DB 부하를 혁신적으로 줄이고 응답 속도를 비약적으로 끌어올림.

### 🔄 모듈 3 대비 구조적 개선점 및 연계 사항
* **비즈니스 정책 레이어 완전 위임**: `friend` 및 `message` 도메인 내에 파편화되어 분산되어 있던 예외 처리와 검증 로직을 도메인 정책 클래스(`EligibilityPolicy`)로 100% 완전 위임하여 비즈니스 응집도를 극대화했습니다.
* **CQRS 기반의 Service 클래스 통합**: 기존에 기능별로 수없이 쪼개져 있어 유지보수가 어렵던 파일들을 Aggregate 기준으로 구조화하여 `Command`와 `Query` 단 두 개의 명확한 서비스 클래스로 통합 리팩토링했습니다.
* **전 가용 채널 웹소켓 연동**: 모듈 3에서 프론트엔드 미구현으로 단방향에 그쳤던 웹소켓 아키텍처를 실시간 채팅방과 알림 인프라 전반에 전면 도입하여 완전한 실시간성을 확보했습니다.

## 2. 해결 과정 및 설계 의도 (Solution & Design)

### 2-1. 하이엔드 성능 최적화: Fan-out 구조 탈피 및 벌크 인메모리 맵 전환
* **AS-IS (기존 방식의 치명적 결함)**: 웹소켓의 실시간 화면 갱신을 처리하기 위해, 채팅방에 참여한 멤버 수만큼 루프(Loop)를 돌며 유저별 메시지 내역 및 채팅방 목록 조회를 매번 개별 쿼리로 실행하는 **Fan-out 구조**였습니다. 이는 대화방 인원수와 데이터 양에 비례하여 SQL 요청이 기하급수적으로 증가($N+1$ 부하)하는 문제를 낳았고, 결국 DB 커넥션 풀 고갈 및 락(Lock) 리스크, API 응답 지연으로 이어졌습니다.
* **TO-BE (아키텍처 개선)**: 반복문 내부의 개별 쿼리 호출을 완전히 전면 제거했습니다. 대신 SQL `IN 절`을 활용하여 필요한 모든 유저의 데이터를 **단 1방의 벌크(Bulk) 쿼리로 일괄 조회**한 뒤, 백엔드 애플리케이션 **메모리(Map)에 적재하여 $O(1)$ 속도로 매핑**하도록 획기적으로 개선했습니다. 이로 인해 동시성 제어 환경에서의 데드락 리스크가 원천 차단되었습니다.

### 2-2. 동기/비동기 관심사 분리 (`@Async` 레이어 격리)
* 사용자에게 즉각 응답이 필요한 핵심 상태 변경(메시지 읽음 처리, 초대 정보 저장 등)은 오직 **동기(Sync) 레이어**에서 신속하게 처리 후 먼저 반환함으로써 체감 레이턴시를 최적화했습니다.
* 무겁고 오버헤드가 큰 웹소켓 Push용 페이로드(Payload) 가공 및 데이터 전송 로직은 스프링의 `@Async` 기반 **비동기(Async) 레이어로 완전 격리**하여 메인 비즈니스 트랜잭션을 안전하게 보호하도록 이원화했습니다.

### 2-3. 모듈 3 결함 보완 및 DTO 반환 구조 리팩토링
* 모듈 3에서 한계로 지적되었던 '방/멤버 정보 유실 결함'을 완전히 해결했습니다. 메시지 이력 테이블만 역순으로 조회하던 한계를 극복하고, 메시지 내역이 없더라도 `ChatRoom` 및 참여자 매핑 테이블을 크로스 체크하여 **상대방의 프로필 정보와 방 메타데이터가 무조건 보장되도록 복합 쿼리 구조로 고도화**했습니다.
* 기존 일대일 채팅방 연계 시 메시지 내역만 역추적하여 메시지 내역 부재 시 무한 신설이 가능한 한계를, 시스템 메시지(안내 문구)까지 교차 추적하도록 개선하여 **기존 일대일 채팅방 연계 정합성을 100% 보장**했습니다.

## 3. 구조 및 데이터 흐름 (Architecture & Data Flow)

1. **Event Trigger**: 메시지 전송, 친구 수락, 알림 삭제 등의 이벤트가 동기 레이어에서 처리 완료되면 비동기 이벤트 헨들러가 구동됩니다.
2. **Bulk Data Fetching**: 루프를 돌며 개별 쿼리를 날리는 대신, 연관된 유저 ID 목록을 취합해 `IN` 절 쿼리 1방으로 필요한 데이터를 일괄 조회합니다.
3. **In-Memory Mapping ($O(1)$)**: 가져온 대량의 엔티티 데이터를 Java `Map<Long, User>` 구조로 메모리에 가공 배치하여, 인원수대로 루프를 순회할 때 DB 추가 조회 없이 메모리 접근만으로 페이로드를 조립합니다.
4. **Multi-Channel WebSocket Push**: 가공된 데이터를 목적에 맞는 정밀 구독 채널로 방출합니다.
    * `/user/sub/notice/total-counts`: 상단 종 모양 실시간 카운트 동기화
    * `/user/sub/notice/list`: 알림 목록 페이지 실시간 리스트 드로우
    * `/user/sub/chat/room/{roomId}`: 채팅방 목록, 메시지 내역 실시간 갱신

## 4. 트러블슈팅 (Troubleshooting)

### 4-1. 프론트엔드 연동 및 API 응답 결함 제어
> **❌ 무분별한 NULL 필드 생략으로 인한 프론트엔드 방어 코드 폭증 이슈**
> * **현상**: 데이터가 비어있을 때(`null`) 필드 자체를 생략하고 반환하자, 프론트엔드 단에서 `undefined` 에러를 막기 위한 방어 코드가 폭증함.
> * **원인**: null 값을 응답 객체에서 무작위로 축소 누락시키는 것이 무조건 깔끔한 응답 규격이라고 오인했던 설계적 미숙함.
> * **해결**: 협의를 통해 값이 없더라도 API 명세 스펙의 구조를 100% 유지하여 빈 객체나 `null` 필드로 형태를 고정해 전달하도록 포맷을 조율하여 클라이언트의 데이터 소비 효율성을 높임.

> **❌ 알림 개수 실시간 미표출 결함 및 책임 공방 이슈**
> * **현상**: 웹소켓 구현 완료 후 알림 개수가 실시간으로 화면에 갱신되지 않는 현상이 발생하여 프론트엔드 측에서 백엔드 로직 조사를 요청함.
> * **원인**: 백엔드 로그 확인 결과 STOMP 데이터는 정상 방출 중이었으나, 프론트엔드 단에서 "알림 개수가 1개 이상일 때만" 화면을 렌더링하도록 조건문이 잘못 설정되어 `0`개일 때의 동기화가 누락됨.
> * **해결**: 정밀 로깅을 통해 결함 위치를 명확히 격리해 증명했고, 프론트엔드 단의 노출 조건문을 `0`개 이상일 때도 작동하도록 함께 수정하여 연동 신뢰성을 확보함.

> **❌ 사용자 검색 응답 데이터 필드(name / nickname) 역전 현상**
> * **현상**: 사용자 검색 응답 데이터 스펙에서 닉네임이 나가야 할 자리에 본명(`name`)이 포맷팅되어 나가는 결함 발생.
> * **원인**: Response DTO 클래스 선언부 및 서비스, UseCase 레이어 전체에서 `name`과 `nickname` 변수의 할당 위치가 서로 뒤바뀌어 정의됨. (이전 빌드 시 CodeRabbit AI 경고를 누락한 건으로 확인)
> * **해결**: DTO 스펙 및 바인딩 순서를 `name`, `nickname` 정방향 순서로 전면 재정렬하여 데이터 정합성 복구.

> **❌ API 명세 규격과 상이한 프론트엔드 독자 가공 및 연동 지연 문제**
> * **현상**: 마감 직전 연동 테스트 중 프론트엔드가 Request Body를 명세와 다르게 전달하거나, 백엔드가 가공을 마친 데이터를 화면단에서 임의로 이중 가공하여 화면이 깨지는 현상 발생. 
> * **원인**: 백엔드 미완성 상태에서 연동을 선행하지 않고 마감 직전에 조급하게 연동을 시작했으며, 기획 스펙(멤버가 1명 남은 퇴장 방을 '나와의 채팅방'으로 오인식 등)을 오인함.
> * **해결**: 백엔드가 정제한 데이터를 그대로 매핑하도록 가이드하고, '나와의 채팅방' 식별 오류는 시스템 가입 시 생성되는 '최초 생성 방 번호(가장 작은 Room ID)'를 고유 기준으로 제공하여 프론트엔드 측의 가공 오류를 강제 교정함.

### 4-2. 데이터베이스(DBA 역할) 및 아키텍처 인프라 트러블 슈팅
> **❌ 소통 부재로 인한 ERD 무단 버전 파편화 리스크**
> * **현상**: 특정 팀원이 요구사항 수정을 이유로 기존 합의된 ERD 외에 임의의 새 버전 ERD 파일들을 파편화하여 생성함.
> * **원인**: 비대면(슬랙) 중심의 텍스트 소통만으로는 인프라 공통 설계를 완벽히 통제하고 공유하기 어려웠던 협업 장벽.
> * **해결**: 즉시 오프라인 대면 회의를 긴급 소집하여 직접 조율을 주도함. 요구사항을 모듈 4 최종본 ERD 단 한 곳으로 완벽히 통합 반영하고, 임의의 구버전 파편화 모델들은 일괄 폐기하여 팀 내 데이터 구조 정합성을 지킴.

> **❌ 다대다 확장 분기로 인한 컬렉션 검증 및 데이터 삽입 에러**
> * **현상**: 다대다(Group) 채팅방 구조로 확장 시, 방 개설 및 조회 과정에서 지속적인 500 에러 및 Insert SQL 에러가 발생함.
> * **원인**: 
>   1. 리스트 형태로 입력되는 초대 대상자 검증에 문자열용 `@NotBlank`를 유지하여 바인딩 에러 발생.
>   2. 다대다 분기 시 `chat_room` 테이블의 변경 이력 추적 컬럼인 `updated_at` 누락으로 DB 제약조건 위배.
>   3. 다대다 방 이름 검증 시 `isEmpty()` 검사를 `null` 체크보다 선행하여 문법적 결함 발생.
> * **해결**: 최소 1개 이상의 요소를 검증하는 `@NotEmpty`로 전환 후 외부 `@Valid` 설정을 제거해 예외 파이프라인을 단일화함. 라이프사이클에 맞춰 `updated_at`을 강제 설정하도록 보완했으며, 조건문 순서를 `null` 체크 선행으로 전면 교정함.

> **❌ 타 도메인 개발자의 알림(Notice) 도메인 무단 침범 및 하드 코딩 충돌 이슈**
> * **현상**: 브랜치 병합 과정에서 게시글(Post) 도메인 담당 팀원이 알림 서비스 내부 레이어에 직접 접근하여 인서트 코드를 무단으로 작성해두어 패키지 및 쿼리 충돌 발생.
> * **원인**: 타 도메인 이벤트 발생 시 비동기 이벤트 핸들링 구조를 거쳐야 하는 인프라적 아키텍처 약속을 누락하여 발생한 격리 실패 결함.
> * **해결**: 무단 머지된 코드를 즉시 격리 처리하고, 게시글 생성 이벤트를 발행하면 알림 도메인의 비동기 리스너가 이를 구독하여 독립적으로 처리하는 방식으로 리팩토링 및 충돌 수습 완료.

### 4-3. AI 도구(CodeRabbit 등) 및 아키텍처 설계 사상에 대한 고찰
> **❌ 무분별한 무지성 AI 코드 리뷰로 인한 기능 오작동 불안 이슈**
> * **현상**: 코드래빗(CodeRabbit) 및 팀원들이 타 AI 툴을 이용해 기획 의도와 맞지 않는 무분별한 리뷰 피드백을 남발하여 개발 착수가 지연되는 병목 발생.
> * **원인**: AI 도구의 특성과 복잡한 웹소켓 도메인의 비즈니스 맥락에 대한 깊은 이해 없이 피드백을 기계적으로 수용하려 했던 태도.
> * **해결**: 리뷰 내용에 종속되지 않고 엔지니어가 직접 코드를 대조해가며 '실제 리팩토링이 필요한 결함'과 '도메인 기획 의도를 고수해야 하는 로직'을 주체적으로 선별 검증하는 필터링 프로세스를 확립함.

> **❌ 클린/헥사고날 아키텍처와 JPA 의존성 간의 유지보수성 딜레마**
> * **현상**: 헥사고날 아키텍처를 지향했으나, 빠른 기능 개발 과정에서 일부 서비스 클래스가 `SpringDataRepository` 및 타 도메인 엔티티 관계를 직접 참조하여 완전한 격리에 실패함.
> * **인사이트**: 다대다 구조 확장 시, 어설픈 격리 구조 때문에 테이블 하나가 바뀌자 서비스부터 유스케이스, 어댑터까지 모든 클래스를 전면 수정해야 하는 시간 낭비가 초래됨.
> * **결론**: 단일 프레임워크를 영속적으로 사용하는 환경에서는 JPA의 강력한 자동 기능과 Fetch Join을 100% 누리는 것이 생산성 면에서 유용할 수 있음을 체감함. 단, 복잡한 통계성 조회나 N+1 방어를 위해서는 **직접 명확한 JPQL/네이티브 쿼리를 수동 작성하는 것이 비즈니스 레이어를 훨씬 가볍게 만드는 지름길**임을 깊이 깨달음.

> * **현상**: 원드라이브 백업 엔진이 동기화 과정에서 로컬 프로젝트의 모든 `.git` 설정 파일 및 디렉토리 매핑 정보를 덮어써 인텔리제이 내 프로젝트 인식이 완전히 박살 나고 브랜치 정보가 소멸함.
> * **원인**: 윈도우 인프라 레벨에서 켜진 원드라이브 백업 엔진이 개발 워크스페이스 내 파일 시스템과 로컬 깃 시스템 디렉토리를 물리적으로 오염시켰기 때문.

### 6-1. 클라이언트(프론트엔드) 연동 및 기획 정합성 제어
> **❌ 클라이언트 단 렌더링 조건 오류로 인한 실시간 알림 누락 결함 제어**
> * **현상**: 웹소켓 인프라 구축 완료 후, 특정 상황에서 알림 개수가 화면에 실시간으로 반영되지 않는 현상이 발생하여 상호 간의 원인 규명이 요구됨.
> * **원인**: 백엔드의 STOMP 주소 발행 및 데이터 정합성 로그를 역추적한 결과 서버 데이터는 정상 방출되고 있었으나, 클라이언트 단에서 "알림 개수가 1개 이상일 때만" 화면을 렌더링하도록 조건문이 잘못 설정되어 있어 `0`개로 수렴할 때의 화면 동기화가 누락된 것으로 확인됨.
> * **해결**: 서버 측 발행 데이터와 디버깅 메트릭을 논리적으로 제시하여 결함의 위치를 명확히 격리했고, 클라이언트 단의 노출 조건문을 `0`개 이상일 때도 정상 반영되도록 협업하여 수정 수습함. 교차 검증을 통해 연동 신뢰성을 확보했습니다.

> **❌ 기획 스펙 해석 차이로 인한 데이터 임의 가공 및 결함 강제 교정**
> * **현상**: 통합 테스트 중 백엔드가 비즈니스 가공을 마친 데이터(나와의 채팅방 식별, 상대가 나간 방 분기 등)를 화면단에서 임의로 이중 가공하여 화면 데이터가 깨지거나 잘못 파싱되는 현상이 발생함.
> * **원인**: 클라이언트 측에서 단일 유저 대화방의 특성(예: 상대방이 퇴장하여 멤버가 1명만 남은 방)을 화면단에서 멤버 수 기준으로 단순 필터링하여 기획된 도메인 스펙과 다르게 오인식함.
> * **해결**: 백엔드가 메시지 및 안내 문구 역추적 시스템을 통해 완벽히 정제한 데이터를 그대로 화면에 배치하도록 데이터 흐름을 재가이드함. 특히 '나와의 채팅방' 식별 오류는 시스템 가입 시 자동 생성되는 '최초 생성 방 식별자(가장 작은 Room ID)'를 고유 기준으로 제공하여 클라이언트 측의 가공 오작동을 구조적으로 교정함.

> **❌ 연동 시점 지연에 따른 API 규격 변동 사항 실시간 동기화**
> * **현상**: 프로젝트 마감 직전 통합 연동이 집중되면서, 초기 설계된 API 명세와 개발 과정에서 고도화된 백엔드 비즈니스 로직(예: 포스트 알림과 방명록 알림의 타입 분기 등) 간의 동기화가 지연되어 인터페이스 불일치 리스크가 발생함.
> * **원인**: 구현 마감 직전에 연동 테스트가 시작되면서 개발 중 변경된 세부 스펙(알림 Type 명세 등)에 대한 실시간 피드백 루프가 원활하지 못했음.
> * **해결**: 클라이언트의 질의에 대해 명확한 비즈니스 구현 결과를 즉각적으로 공유하고, 변경된 응답 데이터 구조와 API 스펙을 명확히 가이드하여 마감 직전의 연동 병목을 해결함.

### 6-2. 데이터베이스(DBA) 및 도메인 격리 트러블 슈팅
> **❌ 형상 관리 경로 분리 부족으로 인한 ERD 모델 파편화 리스크 수호**
> * **현상**: 개발 속도가 가속화되는 과정에서 일부 팀원이 요구사항 수정을 반영하기 위해 기존 합의된 ERD 외에 별도의 변형된 모델 파일들을 파편화하여 생성함.
> * **원인**: 비대면 중심의 텍스트 소통만으로는 복잡한 변경 사항과 인프라 공통 설계를 전 팀원에게 실시간으로 완벽하게 동기화하기 어려웠던 고질적인 장벽.
> * **해결**: 텍스트 소통의 한계를 인지하고 대면 회의를 긴급 소집하여 직접 조율을 주도함. 산재된 모든 요구사항을 모듈 4 최종본 ERD 단 한 곳으로 완벽히 통합 반영하고 파편화된 구버전 모델들을 정리하여, 팀 내 데이터 아키텍처 리더십(DBA 역할)을 확고히 수행함.

> **❌ 바운디드 컨텍스트(Bounded Context) 경계 모호로 인한 도메인 레이어 침범 및 충돌 수습**
> * **현상**: 브랜치 병합 과정에서 타 도메인(Post) 담당 개발자가 알림(Notice) 서비스 내부 레이어에 직접 접근하여 엔티티 제어 및 인서트 코드를 작성해두어 패키지 및 쿼리 충돌이 발생함.
> * **원인**: 타 도메인에서 알림 발생 시 이벤트 발행 후 비동기 핸들링 구조를 거쳐야 하는 인프라적 아키텍처 약속과 도메인 간 경계 격리에 대한 공유가 미흡하여 발생한 결함.
> * **해결**: 해당 코드를 즉시 격리 처리하고, 전체 공지를 통해 게시글 생성 이벤트를 발행하면 알림 도메인 내의 비동기 리스너가 이를 구독하여 독립적으로 처리하는 구조로 리팩토링 및 충돌 수습을 완료함.

> **❌ 도메인 비즈니스 규칙이 배제된 대량 데이터셋(더미 데이터) 오염 대응**
> * **현상**: 외부에서 대량의 더미 데이터가 유입되는 과정에서 본인이 담당한 도메인의 핵심 비즈니스 규칙(회원가입 시 나와의 채팅방 자동 개설, 강사-학생 자동 친구 맺기, 안내 문구 생성 로직 등)이 고려되지 않아 시스템 유효성 검증 시 데이터 불일치 에러가 발생함.
> * **원인**: 각 도메인의 세부 라이프사이클 규칙과 연동 구조를 완벽히 파악하지 못한 채 인프라 레벨에서 데이터가 일괄 투입되었기 때문.
> * **해결**: 전체 유실 데이터를 수동으로 수정하기에는 일정이 촉박함을 인지하고, 서비스 가동에 치명적인 영향을 줄 수 있는 '나와의 채팅방 방 번호 규칙' 등 핵심 트리거 위주로 데이터를 정밀 수선함. 이를 통해 대규모 부하 테스트 및 배포 환경에서 예외가 터지지 않도록 데이터 안정성을 확보함.

### 6-3. 통합 배포 환경에서의 기술적 신뢰성 보증
> **❌ 통합 배포 환경에서의 웹소켓 인증 메커니즘 변동 및 인프라 머지 거부 돌파**
> * **현상**: 성능 개선 및 웹소켓 안정화 코드를 반영한 브랜치를 배포용 메인 브랜치에 병합하려 했으나, 변경된 웹소켓 토큰 프로바이더 로직이 타 기능에 영향을 줄 수 있다는 팀원들의 우려로 인해 머지가 보류되는 병목이 발생함.
> * **원인**: 로컬 환경과 실제 배포(클라우드) 환경에서의 웹소켓 인증 방식 차이에 대한 이해 부족 및 사이드 이펙트에 대한 팀 내 막연한 불안감.
> * **해결**: 감정적 소통을 지양하고, 배포용 격리 브랜치에 내 성능 개선 브랜치를 직접 연결하여 사전 배포 테스트를 주도함. 타 기능이 100% 정상 작동함을 논리적 데이터로 증명하여 팀원들의 우려를 불식시키고 성공적으로 코드를 전면 머지함. 배포 후 발생한 세부 인증 결함은 클라이언트 배포 환경과 소통하며 수정 코드를 핀포인트로 전달해 실시간으로 수습 완료함.

### 6-4. AI 도구 및 코드 리뷰 수용 프로세스 정립
> **❌ 생성형 AI 피드백의 무분별한 교차 리뷰로 인한 개발 병목 제어**
> * **현상**: 팀 내부적으로 다양한 생성형 AI 툴을 활용한 코드 리뷰 지적 사항들이 쏟아지면서, 이미 정상 작동하는 기능마저 기획 의도와 다르게 변질되거나 개발 착수가 지연되는 소통 피로가 발생함.
> * **원인**: 복잡한 웹소켓 도메인의 흐름과 기획 요구사항에 대한 깊은 맥락 이해 없이 도구의 피드백을 기계적으로 수용하려 했던 팀 내 접근 방식의 한계.
> * **해결**: 도구의 제안에 종속되지 않고 엔지니어가 직접 코드를 대조해가며 '실제 리팩토링이 필요한 결함'과 '도메인 기획 의도를 고수해야 하는 로직'을 주체적으로 선별 검증하는 필터링 프로세스를 수립하여 팀의 코드 안정성을 방어함.

## 7. 한계 및 향후 개선 과제 (Limitations & Next Steps)

* **웹소켓 통신 데이터 부하 최적화 한계**: 현재 응답 데이터 규격 통일을 목적으로 CUD(생성/수정/삭제) 발생 시 웹소켓 조회 서비스를 직접 호출하여 무거운 페이로드 전체를 전송하고 있습니다. 이는 사용자 급증 시 잠재적인 네트워크 대역폭 폭탄이 될 수 있으므로, 향후에는 대용량 페이로드 대신 **"갱신이 필요하다는 최소한의 식별 시그널(Notice ID 등)만 전송"**하고 클라이언트가 이를 받아 필요한 부분만 핀포인트로 가져가도록 경량화 아키텍처로 개선할 예정입니다.
* **도메인 간 결합도 제어(구조적 한계)**: 현재 일부 서비스 레이어가 연관 데이터 처리를 위해 타 도메인의 `JpaEntity` 객체와 강하게 결합되어 있습니다. 향후 바운디드 컨텍스트(Bounded Context) 간의 완벽한 격리를 위해, 엔티티 객체 참조를 과감히 제거하고 오직 **식별자(Long Id) 기반 참조** 및 이벤트 발행 구조로 전환하여 시스템 확장성을 고도화할 계획입니다.
