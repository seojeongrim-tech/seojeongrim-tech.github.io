# ⚙️ 실시간 웹소켓 인프라 및 대규모 데이터 최적화기 (Module 4)

> **핵심 요약:** Fan-out 구조 개선을 통한 DB 커넥션 고갈 및 락(Lock) 리스크 원천 차단, 동기/비동기 관심사 분리를 통한 메인 트랜잭션 보호, IN 절 활용 벌크(Bulk) 조회를 통한 $O(1)$ 인메모리 매핑 혁신, 화면 레이턴시 최대 57% 단축 및 프론트엔드 연동 결함 격리

---

## 1. 개요
모듈 4에서는 앞서 구축한 친구 및 메시징 도메인을 확장하여, 시스템 전반의 실시간 커뮤니케이션을 담당하는 **실시간 웹소켓(STOMP) 알림 인프라**와 **다중 채널 실시간 갱신 시스템**을 전담하여 구현했습니다. 특히 대규모 트래픽 환경에서 발생할 수 있는 데이터베이스 병목을 해결하기 위해 백엔드 아키텍처를 전면 리팩토링하였으며, 모듈 4에서도 **DBA 역할**을 연속 수행하여 팀 내 인프라 결함과 파편화 문제를 주도적으로 돌파했습니다.

---

### 👥 팀 구성 및 협업 환경
* **팀 구성**: **백엔드 5명 (JSON 응답 설계) / 프론트엔드 2명**
* **담당 역할**: 웹소켓(STOMP) 실시간 알림 인프라 구축, 알림 라이프사이클 최적화, 대규모 조회 성능 개선 및 프로젝트 DBA 역할 리드
* **핵심 성과**: 단순 기능 구현을 넘어, 웹소켓 실시간 화면 갱신 시 발생하는 Fan-out 문제를 인메모리 벌크 아키텍처로 개선하여 DB 부하를 혁신적으로 줄이고 응답 속도를 비약적으로 끌어올림.

## 2. 해결 과정 및 설계 의도 (Solution & Design)

### 2-1. 하이엔드 성능 최적화: Fan-out 구조 탈피 및 벌크 인메모리 맵 전환
* **AS-IS (기존 방식의 치명적 결함)**: 웹소켓의 실시간 화면 갱신을 처리하기 위해, 채팅방에 참여한 멤버 수만큼 루프(Loop)를 돌며 유저별 메시지 내역 및 채팅방 목록 조회를 매번 개별 쿼리로 실행하는 **Fan-out 구조**였습니다. 이는 대화방 인원수와 데이터 양에 비례하여 SQL 요청이 기하급수적으로 증가($N+1$ 부하)하는 문제를 낳았고, 결국 DB 커넥션 풀 고갈 및 락(Lock) 리스크, API 응답 지연으로 이어졌습니다.
* **TO-BE (아키텍처 개선)**: 반복문 내부의 개별 쿼리 호출을 완전히 전면 제거했습니다. 대신 SQL `IN 절`을 활용하여 필요한 모든 유저의 데이터를 **단 1방의 벌크(Bulk) 쿼리로 일괄 조회**한 뒤, 백엔드 애플리케이션 **메모리(Map)에 적재하여 $O(1)$ 속도로 매핑**하도록 획기적으로 개선했습니다. 이로 인해 동시성 제어 환경에서의 데드락 리스크가 원천 차단되었습니다.

### 2-2. 동기/비동기 관심사 분리 (`@Async` 레이어 격리)
* 사용자에게 즉각 응답이 필요한 핵심 상태 변경(메시지 읽음 처리, 초대 정보 저장 등)은 오직 **동기(Sync) 레이어**에서 신속하게 처리 후 먼저 반환함으로써 체감 레이턴시를 최적화했습니다.
* 무겁고 오버헤드가 큰 웹소켓 Push용 페이로드(Payload) 가공 및 데이터 전송 로직은 스프링의 `@Async` 기반 **비동기(Async) 레이어로 완전 격리**하여 메인 비즈니스 트랜잭션을 안전하게 보호하도록 이원화했습니다.

### 2-3. 모듈 3 결함 보완 및 DTO 반환 구조 리팩토링
* 모듈 3에서 한계로 지적되었던 '연속 발신 시 방 정보 유실 결함'을 완전히 해결했습니다. 메시지 이력 테이블만 역순으로 조회하던 한계를 극복하고, 최근 내역이 전부 본인이 작성한 메시지이더라도 `ChatRoom` 및 참여자 매핑 테이블을 크로스 체크하여 **상대방의 프로필 정보와 방 메타데이터가 무조건 보장되도록 복합 쿼리 구조로 고도화**했습니다.

## 3. 구조 및 데이터 흐름 (Architecture & Data Flow)

1. **Event Trigger**: 메시지 전송, 친구 수락, 알림 삭제 등의 이벤트가 동기 레이어에서 처리 완료되면 비동기 이벤트 헨들러가 구동됩니다.
2. **Bulk Data Fetching**: 루프를 돌며 개별 쿼리를 날리는 대신, 연관된 유저 ID 목록을 취합해 `IN` 절 쿼리 1방으로 필요한 데이터를 일괄 조회합니다.
3. **In-Memory Mapping ($O(1)$)**: 가져온 대량의 엔티티 데이터를 Java `Map<Long, User>` 구조로 메모리에 가공 배치하여, 인원수대로 루프를 순회할 때 DB 추가 조회 없이 메모리 접근만으로 페이로드를 조립합니다.
4. **Multi-Channel WebSocket Push**: 가공된 데이터를 목적에 맞는 정밀 구독 채널로 방출합니다.
    * `/user/sub/notice/total-counts`: 상단 종 모양 실시간 카운트 동기화
    * `/user/sub/notice/list`: 알림 목록 페이지 실시간 리스트 드로우

## 4. 트러블슈팅 (Troubleshooting)

> **❌ 무분별한 NULL 필드 생략으로 인한 프론트엔드 방어 코드 폭증 이슈**
> * **현상**: 초기 백엔드 응답 설계 시 데이터가 비어있을 때(`null`) 필드 자체를 생략하고 내보내자, 프론트엔드 단에서 `undefined` 에러를 막기 위해 일일이 수많은 렌더링 방어 코드를 짜야 하는 병목이 발생함.
> * **원인**: null 값을 API 응답 객체에서 무작위로 축소 누락시키는 것이 무조건 깔끔한 응답 규격이라고 오인했던 설계적 미숙함.
> * **해결**: 프론트엔드와의 긴밀한 재정비 협의를 통해, 값이 없더라도 API 명세 스펙의 구조를 100% 유지하여 빈 객체나 `null` 필드로 형태를 고정해 전달하도록 포맷을 전면 조율함. 클라이언트의 데이터 소비 효율성을 최대로 극대화했습니다.

> **❌ 알림 개수 실시간 미표출 결함 및 책임 공방 이슈**
> * **현상**: 웹소켓 구현 완료 후 알림 개수가 실시간으로 화면에 갱신되지 않는 치명적 현상이 발생, 프론트엔드 측에서는 구독 주소와 백엔드 로직 문제라며 전면 확인을 요청함.
> * **원인**: 백엔드의 STOMP 주소 발행 및 정합성 로그를 추적한 결과, 데이터는 정상 방출되고 있었음. 실제 원인은 프론트엔드 측에서 "알림 개수가 1개 이상일 때만 화면에 그리도록" 조건문을 잘못 설정해 두어 `0`개일 때 갱신이 누락되었던 것임.
> * **해결**: 정밀 로깅을 통해 결함의 위치를 명확히 백엔드 외부로 격리해 증명했고, 프론트엔드 단의 조건문을 `0`개 이상일 때도 정상 노출되도록 함께 코드를 수정하여 해결함. 소통의 한계를 극복하고 연동 신뢰성을 확보했습니다.

> **❌ 소통 부재로 인한 ERD 무단 버전 파편화 리스크**
> * **현상**: 개발이 한창 진행되는 도중, 특정 팀원이 요구사항 수정을 이유로 기존 합의된 ERD를 무시하고 임의의 새 버전 ERD를 따로 파편화시켜 생성함.
> * **원인**: 비대면(슬랙) 중심의 텍스트 소통만으로는 수많은 변경사항และ 인프라 공통 설계를 완벽히 통제하기 어려워 발생하는 고질적인 팀 커뮤니티 장벽.
> * **해결**: 슬랙의 한계를 인지하고 즉시 오프라인 및 구두 소통을 주도하여 회의를 긴급 소집함. 모든 요구사항을 모듈 4 최종본 ERD 단 한 곳으로 완벽히 통합 반영하고, 독단적으로 생성되었던 임의의 구버전 파편화 모델들은 일괄 폐기 처리하여 팀 내 데이터 아키텍처 리더십(DBA 역할)을 확고히 수행함.

> **❌ 무분별한 무지성 AI 코드 리뷰로 인한 기능 오작동 불안 이슈**
> * **현상**: 코드래빗(CodeRabbit) 및 팀원들이 타 AI 툴을 이용하여 기획 의도와 전혀 맞지 않는 무분별한 코드 리뷰 지적을 남발하여, 되던 기능마저 꼬이고 다른 개발에 착수하지 못하는 병목 현상 발생.
> * **원인**: AI 도구의 특성과 복잡한 웹소켓 도메인에 대한 깊은 이해 없이 기계적인 알림 리뷰 피드백을 무지성으로 전면 수용하려 했던 태도.
> * **해결**: 리뷰 내용에 무조건 휘둘리지 않고, 주체적으로 눈으로 직접 코드를 대조해가며 '진짜 수정해야 하는 로직'과 '내 도메인 기획 의도에 완벽히 부합하는 로직'을 선별 검증하는 필터링 프로세스를 확립하여 코드 안정성을 방어함.

## 6. 팀 협업 및 소통의 한계 극복 (Team Collaboration & Hardships)

모듈 4 스프린트 역시 파편화된 팀 환경과 프론트엔드의 비협조적인 태도로 인해 수많은 심리적 한계와 마주해야 했습니다.

* **"저는 모르죠;;" 장벽을 넘어선 집념**: 웹소켓 연동 중 발생한 버그에 대해 대뜸 백엔드 탓을 하며 방관하는 프론트엔드 개발자의 태도 앞에서도 감정적으로 위축되지 않았습니다. 철저한 백엔드 로그 모니터링과 아키텍처 정합성 증명을 통해 결함의 실체(프론트 단 조건문 오류)를 직접 짚어내어 결국 상대를 납득시키고 기능을 정상화했습니다.
* **DBA로서의 묵묵한 중심 잡기**: 슬랙으로 아무리 ERD 가이드라인을 외치고 공유해도 팀원들이 적극적으로 듣지 않고 임의로 테이블을 파편화시키는 무력한 상황 속에서, 텍스트 소통을 과감히 멈추고 **직접 대면하여 구두로 조율하는 직접 돌파형 소통**으로 팀의 데이터 무결성을 끝까지 수호했습니다.
* **철저한 예외 방어가 만든 완벽한 연동**: 수많은 AI 코드 리뷰 공세와 소통 피로감 속에서도 비즈니스 예외 레이어(4단계 검증, 중복 연산 원천 제외)를 촘촘하게 독자적으로 밀어붙인 결과, 악조건의 연동 환경 속에서도 단 하나의 데이터 오염이나 레이스 컨디션 없이 실시간 웹소켓 인프라를 성공적으로 안착시키며 백엔드 엔지니어로서의 강력한 기술적 품질을 증명해냈습니다.
