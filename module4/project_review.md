# ⚙️ 맡은 기능 (Module 4)

모듈 3에서 구축한 친구 및 메시징 도메인을 확장하여, 시스템 전반의 실시간 커뮤니케이션을 담당하는 **실시간 웹소켓(STOMP) 알림 인프라**와 **다중 채널 실시간 갱신 시스템**을 전담하여 구현했습니다. 실시간 데이터 정합성을 유지하기 위해 수많은 예외 케이스와 비즈니스 정책을 촘촘하게 방어했으며, 모듈 4에서도 **DBA 역할**을 주도적으로 수행하여 팀 내 인프라 결함을 돌파했습니다.

#### 실시간 알림 인프라 및 예외 방어 혁신
> * **다중 채널 웹소켓 설계**: 화면의 UI 요소(상단 종 모양, 개별 앱 아이콘, 알림 목록 페이지)에 맞춰 필요한 데이터만 정밀하게 갱신하는 멀티 구독 채널 아키텍처 정립.
> * **철저한 예외 검증 레이어**: 사용자가 알림을 읽거나 조작할 때 발생할 수 있는 데이터 부재, 권한 위반, 중복 자원 낭비를 예외 케이스 기반으로 촘촘하게 방어.
> * **주도적 DB 설계 리드**: 팀 내 ERD 파편화 고질병을 주도적으로 수정.

## 1. 멀티 채널 실시간 웹소켓(STOMP) 알림 인프라 구축

웹소켓 구독 채널을 정밀하게 분리하고 데이터를 실시간 가공하여 전송하는 백엔드 시스템을 구축했습니다.

* **종 모양 실시간 총량 갱신 (`/user/sub/notice/total-counts`)**:
    * 사용자가 알림 목록에서 가공한 메시지 관련 알림 행 수(isNotiRead 및 isDeleted가 false인 채팅방 개수)를 `message_read` 테이블에서 실시간 집계.
    * 일반 알림이 새로 발생하거나, 기존 알림을 읽고 삭제할 때마다 해당 구독 채널로 최신 총량을 즉시 push하여 종 모양 아이콘 숫자가 실시간 동기화되도록 구현.
* **앱별 알림 개수 다중 연산 인프라**:
    * 휴대폰 홈 화면 스타일의 UI를 지원하기 위해, 로그인 유저의 읽지 않은 메시지 개수(`isMsgRead = false`)와 친구 관련 알림 개수를 합산하는 MESSAGE 타입 알림 연산 로직 구현.
    * 캘린더(`CALENDAR`) 및 커뮤니티(`POST`) 등 타 도메인에서 나에게 온 알림 개수까지 실시간으로 계산하여 웹소켓 채널로 방출.
* **알림 목록 동적 갱신 채널 (`/user/sub/notice/list`)**:
    * 알림 목록 페이지를 열어둔 사용자들을 위해 정렬 상태와 읽음 여부가 반영된 리스트 데이터를 실시간으로 동기화 처리.

## 2. 완벽한 정합성을 위한 알림 라이프사이클 및 예외 처리

사용자가 알림 목록에서 일괄 읽음을 수행하거나 특정 알림을 클릭해 이동할 때, 데이터 오염과 무분별한 자원 낭비를 막기 위해 촘촘한 예외 처리 구조를 확립했습니다.

* **데이터 무결성을 위한 4단계 예외 검증 레이어**:
    * *읽을 알림을 선택하지 않고 요청 시* ➡️ `"읽음 처리할 알림이 선택되지 않았습니다."` (HTTP 400 방어)
    * *요청된 알림 ID가 DB에 존재하지 않을 때* ➡️ `"존재하지 않는 알림이 포함되어 있습니다."`
    * *타인의 알림에 접근하거나 읽으려 시도할 때* ➡️ `"해당 알림에 접근할 권한이 없습니다."` (`userId`와 로그인 유저 철저 비교 방어)
    * *자원 절약 및 중복 연산 방지* ➡️ 이미 읽음 처리가 완료된 알림 데이터는 업데이트 대상에서 원천 제외하여 데이터베이스 자원 소모 최소화.
* **도메인 특성을 고려한 읽음 처리 분기**:
    * 일반 알림은 단순 `notification` 테이블의 `userId` 검증만 수행.
    * 메시지 알림의 경우, `notification` 테이블의 참조 ID(`refId`)와 메시지 읽음 상태 테이블(`message_read`)의 방 번호(`roomId`)가 일치하면서 로그인 유저에 해당하는 데이터만 핀포인트로 찾아내 수정 처리하는 정밀 쿼리 구조 설계.

## 3. 프론트엔드 고려 및 상호 커뮤니케이션 중심 트러블 슈팅

* **응답 정합성 협의**: 모듈 4 재정비 과정에서 데이터가 `null`일 경우 응답 JSON에서 아예 생략하는 방식이 프론트엔드 단에서 일일이 null 체크 로직을 짜야 하는 불편함을 유발함을 인지. 프론트엔드가 데이터를 안전하게 소비할 수 있도록 값이 없더라도 필드 구조를 유지하여 전달하는 방식으로 조율.
* **웹소켓 실시간 미표출 결함 격리**: 알림 개수가 실시간으로 반영되지 않는 이슈가 발생했을 때, 백엔드 웹소켓 발행 주소와 정합성을 로그를 기반으로 역추적하여 데이터가 정상 방출됨을 증명. 이후 프론트엔드 단에서 "알림이 1개 초과일 때만 화면에 그리도록 한 조건문 설정 오류"를 찾아내어 결함의 위치를 명확히 격리하고 해결을 지원함.

## 4. DBA 역할 연속 수행 및 DDL 선제 최적화

팀원들의 소통 지연과 인프라 파편화 상황 속에서, 데이터베이스 설계 리더십을 발휘하여 개발 병목을 돌파했습니다.

* **파편화된 ERD 단일화 및 통합 제어**: 모듈 4 재정비 중 팀원이 요구사항을 임의로 반영해 ERD 버전을 무분별하게 새로 파편화시키는 리스크 포착. 슬랙 논의의 한계를 인지하고 직접 구두 소통을 주도하여 모든 요구사항을 통합한 모듈 4 최종본 ERD로 단일화 완료 후 파편화된 구버전은 과감히 폐기 처리.
* **성능을 고려한 선제적 DDL 수립**: 타 파트의 인프라 확정이나 소통 지연을 기다리지 않고, 회의 내용을 바탕으로 혼자 명확한 DDL 구문을 선제적으로 구축하여 개발 속도를 극대화. 특히 DDL 작성 단계에서부터 빠른 검색과 정렬이 가능하도록 **인덱스(KEY) 설계를 미리 고려하여 반영**함으로써 조회 성능 최적화의 기반을 마련.
