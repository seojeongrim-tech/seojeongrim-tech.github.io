# ⚙️ 맡은 기능 (Module 4)

모듈 3에서 구축한 친구 및 메시징 도메인을 확장하여, 시스템 전반의 실시간 커뮤니케이션을 담당하는 **실시간 웹소켓(STOMP) 알림 인프라**와 **다중 채널 실시간 갱신 시스템**을 구현했습니다. 특히 모듈 4에서는 기존 1:1 채팅 인프라를 **다대다(Multi-User) 채팅방 및 테이블 구조로 전면 확장**하며 발생한 예외 정책을 정립했고, 새롭게 도입된 **방명록(Guestbook) 도메인**의 권한 예외 방어선까지 전담했습니다. 실시간 데이터 정합성을 유지하기 위해 수많은 예외 케이스와 비즈니스 정책을 촘촘하게 방어했으며, **DBA 역할**을 주도적으로 수행하여 팀 내 인프라 결함을 돌파했습니다.

---

### 👥 팀 구성 및 협업 환경
* **팀 구성**: 백엔드 5명 (JSON 응답 설계) / 프론트엔드 2명
* **담당 역할**: 웹소켓(STOMP) 실시간 알림 인프라 구축, 다대다 채팅방 확장 및 메시지 로직 전면 수정, 신규 방명록 도메인 개발, 프로젝트 DBA 역할 수행
* **협업 도구**: GitHub, Slack, Notion
* **핵심 타깃**: 단순 조회가 아닌 실시간 웹소켓 구독 환경 및 다대다 채팅 구조에서 다수의 알림·메시지 상태가 꼬이지 않고 완벽한 데이터 정합성을 유지하는 백엔드 인프라 완성

#### 핵심 구현 및 예외 방어 혁신
> * **다중 채널 웹소켓 설계**: 화면의 UI 요소(상단 종 모양, 개별 앱 아이콘, 알림 목록 페이지, 채팅 관련) 갱신을 목적으로 응답 데이터 통일하여 멀티 구독 채널 아키텍처 정립.
> * **다대다 패러다임 전환**: 1:1 구조에서 다대다 확장 및 테이블 구조 개편하고, 사용자 상태 변화를 타임라인에 실시간 추적 반영하는 자동 보정 로직 구현.
> * **철저한 예외 검증 레이어**: 알림, 메시지, 방명록 조작 시 발생할 수 있는 데이터 부재, 권한 위반, 중복 자원 낭비를 예외 케이스 기반으로 촘촘하게 방어.
> * **주도적 DB 설계 리드**: 팀 내 ERD 파편화 고질병을 주도적으로 수정하고 임의 수정에도 ERD과 컬럼 하나씩 비교하며 DDL 선제 수립.

## 1. 멀티 채널 실시간 웹소켓(STOMP) 알림 인프라 구축

웹소켓 구독 채널을 정밀하게 분리하고 데이터를 실시간 가공하여 전송하는 백엔드 시스템을 구축했습니다.

* **종 모양 실시간 총량 갱신 (`/user/sub/notice/total-counts`)**:
    * 사용자가 알림 목록에서 가공한 메시지 관련 알림 행 수(`isNotiRead` 및 `isDeleted`가 `false`인 채팅방 개수)를 `message_read` 테이블에서 실시간 집계.
    * 일반 알림이 새로 발생하거나, 기존 알림을 읽고 삭제할 때마다 해당 구독 채널을 재호출하여 종 모양 아이콘 숫자가 실시간 동기화되도록 구현.
* **앱별 알림 개수 다중 연산 인프라**:
    * 휴대폰 홈 화면 스타일의 UI를 지원하기 위해, 로그인 유저의 읽지 않은 메시지 개수(`isMsgRead = false`)와 친구 관련 알림 개수를 합산하는 MESSAGE 타입 알림 연산 로직 구현.
    * 캘린더(`CALENDAR`) 및 커뮤니티(`POST`) 등 타 도메인에서 나에게 온 알림 개수까지 실시간으로 계산하여 웹소켓 채널로 방출.
* **알림 목록 동적 갱신 채널 (`/user/sub/notice/list`)**:
    * 알림 목록 페이지를 열어둔 사용자들을 위해 정렬 상태와 읽음 여부가 반영된 리스트 데이터를 실시간으로 동기화 처리.

## 2. 1:1 ➡️ 다대다 확장 전환에 따른 채팅 및 메시지 가공 수정 & 채팅 웹소켓(STOMP) 적용

채팅방 구조가 다중 참여 방식으로 확장됨에 따라 유저 상태 변화와 메시지 발신 타이밍이 꼬이지 않도록 촘촘한 비즈니스 가공 정책을 반영했습니다.

* **타 도메인 상태 동기화 및 공통 가공 처리**:
    * 차단 친구 목록, 사용자 검색, 친구 요청 목록 조회 시 도메인별로 개별 가공하던 비활성 유저 예외 처리를 User 담당자가 공통 가공(`ACTIVE`가 아닐 때)하여 제공하도록 협의를 이끌어내 중복 로직 제거.
    * **목록 조회 제어**: 유저 상태가 `ACTIVE`이면서 동시에 나와 친구 상태가 아닌 경우에는 프라이버시 보호 정책에 따라 닉네임을 **`(알 수 없음)`**으로 강제 보정 처리함.
* **채팅 웹소켓(STOMP) 실시간 연동 및 가공 아키텍처**:
    * **채팅방 내부 실시간 메시지 구독 및 발행 채널 (`/sub/chat/room/{roomId}`)**: 유저가 특정 방에 진입 시 해당 채널을 구독하고, 메시지 발신 시 실시간으로 대화가 전달되도록 구현.
    * 발신 유저의 상태를 실시간 역추적하여 **친구 관계가 아니거나 차단(`BLOCK`) 상태로 변경된 유저**가 메시지를 보내고 방을 나간 경우(`isLeftRoom = true`), 응답 값의 닉네임을 자동으로 **`닉네임(알 수 없음)`**으로 변환 결합하여 사용자 편의성을 고려함.
* **다대다 확장에서도 다중 예외 처리**
    * **채팅방 조회 및 개설**:
       * *초대 대상자를 아무것도 안 넘길 경우* ➡️ `"채팅방 멤버는 최소 1명 이상 지정해야 합니다."`
       * *다대다 개설 불가한 사용자 포함 시* ➡️ `"대화창을 개설할 수 없는 사용자가 포함되어 있습니다."` (비학생, 비활성 유저 포함)
       * *로그인 유저 포함* ➡️ `"자기 자신과는 대화창을 개설할 수 없습니다."` (회원가입 시 나와의 채팅방이 개설되므로 정합성 고려)
       * *방 제목 빈 값* ➡️ `"방 제목은 공백일 수 없습니다."`
       * *방 제목 20자 제한* ➡️ `"방 제목은 최대 20자까지 가능합니다."`
       * *일대일 개설 시 방제목 지정 불가* ➡️ `"일대일 채팅방은 방 제목을 지정할 수 없습니다."`
       * *초대 대상자 중복 선택* ➡️ `"동일한 사용자를 중복하여 개설할 수 없습니다."`
  * **방 제목 수정**:
       * *채팅방 미존재* ➡️ `"존재하지 않는 채팅방입니다."`
       * *채팅방에 속한 멤버가 아닐 때* ➡️ `"해당 채팅방에 접근할 권한이 없습니다."`
       * *다대다인 경우에만 설정 가능* ➡️ `"해당 채팅방은 채팅방 이름을 설정할 수 없습니다.(단체 채팅방에 한해서 수정 가능)"` (일대일과 다대다 채팅을 구분하기 위해 프론트와 협의한 내용)
       * *방 제목 빈 값* ➡️ `"방 제목은 공백일 수 없습니다."`
       * *방 제목 20자 제한* ➡️ `"방 제목은 최대 20자까지 가능합니다."`
       * *기존 방 제목과 동일한 경우* ➡️ `"변경 사항이 없어 수정되지 않습니다."` (동일한 값으로 업데이트되는 것을 방지하여 자원 절약.)
  * **채팅방 멤버 초대**:
       * *빈 값 입력* ➡️ `"초대할 대상을 선택해주세요."`
       * *채팅방 미존재* ➡️ `"존재하지 않는 채팅방입니다."`
       * *초대 대상자 미존재* ➡️ `"존재하지 않는 사용자가 포함되어 있어 초대할 수 없습니다."`
       * *채팅방에 속한 멤버가 아닐 때* ➡️ `"해당 채팅방에 접근할 권한이 없습니다."`
       * *일대일 채팅방에서 초대 시도* ➡️ `"채팅방 멤버 초대는 단체 채팅방에서만 가능합니다."`
       * *이미 방에 속한 멤버* ➡️ `"이미 방에 참여중인 멤버가 포함되어 있습니다."`
       * *본인 초대* ➡️ `"자기 자신은 초대할 수 없습니다."` (이미 속한 멤버에서 예외 처리되지만 이중 예외 처리)
       * *초대 불가한 사용자 포함 시* ➡️ `"초대할 수 없는 사용자가 포함되어 있습니다."` (비학생, 비활성 유저 포함)
       * *친구 상태가 아닌 경우* ➡️ `"친구 상태인 사용자만 초대할 수 있습니다."` (초대자와 대상자 사이의 관계에 한함)

## 3. 완벽한 정합성을 위한 알림 라이프사이클 및 예외 처리

사용자가 알림 목록에서 일괄 읽음을 수행하거나 특정 알림을 클릭해 이동할 때, 데이터 오염과 무분별한 자원 낭비를 막기 위해 촘촘한 예외 처리 구조를 확립했습니다.

* **데이터 무결성을 위한 4단계 예외 검증 레이어**:
    * *읽을 알림을 선택하지 않고 요청 시* ➡️ `"읽음 처리할 알림이 선택되지 않았습니다."` (HTTP 400 방어)
    * *요청된 알림 ID가 DB에 존재하지 않을 때* ➡️ `"존재하지 않는 알림이 포함되어 있습니다."`
    * *타인의 알림에 접근하거나 읽으려 시도할 때* ➡️ `"해당 알림에 접근할 권한이 없습니다."` (`userId`와 로그인 유저 철저 비교 방어)
    * *자원 절약 및 중복 연산 방지*: 중복 알림 발생 시 무분별하게 행을 추가하지 않고 하나로 합산 처리하며, 이미 읽음(`isRead = true`) 또는 삭제 처리된 알림 데이터는 업데이트 대상에서 원천 제외하여 DB 자원 소모 최소화 및 예외 처리를 하지 않고 내부적으로 처리하여 사용자 편의성 고려.(알림 삭제의 경우도 동일하게 처리)
* **도메인 특성을 고려한 읽음 처리 분기**:
    * 일반 알림은 단순 `notification` 테이블의 `userId` 검증만 수행.
    * 메시지 알림의 경우, `notification` 테이블의 참조 ID(`refId`)와 메시지 읽음 상태 테이블(`message_read`)의 방 번호(`roomId`)가 일치하면서 로그인 유저에 해당하는 데이터만 핀포인트로 찾아내 수정 처리하는 정밀 쿼리 구조 설계.

## 4. 신규 도입 도메인: 방명록(Guestbook) 권한 및 예외 방어선 구축

이번 모듈 4에서 새롭게 진행된 방명록 기능으로, 사용자의 프로필 공간에 흔적을 남길 때 발생할 수 있는 비정상적인 접근과 권한 침해 유스케이스를 철저히 통제했습니다.

* **방명록 작성 단계의 비즈니스 검증**:
    * 본인의 프로필 공간에 스스로 방명록 글을 남기려는 행위 원천 차단 ➡️ `"본인의 도시에는 방명록을 작성할 수 없습니다."`
    * 친구 상태가 아닐 시 ➡️ `"친구 상태에서만 방명록을 작성할 수 있습니다."`
    * 포인트 제도 악용 방어 ➡️ `"방명록은 하루에 한 번 작성할 수 있습니다."`

## 5. 프론트엔드 고려 및 상호 커뮤니케이션 중심 트러블 슈팅

* **데이터 누락 없는 구조적 JSON 응답 스펙 설계**: 
    * 모듈 4 재정비 과정에서 데이터가 비어있을 때(`null`) 필드 자체를 생략해 버리면 프론트엔드 단에서 `undefined` 에러 방지를 위해 일일이 방어 코드를 짜야 하는 불편함을 유발함을 인지. 
    * 값이 없더라도 명세서 구조를 100% 유지하여 빈 객체나 `null`로 필드를 고정 전달하는 방식으로 API 포맷을 전면 조율하여 프론트엔드의 데이터 소비 효율성 극대화.
* **웹소켓 실시간 미표출 결함 격리**: 
    * 알림 개수가 실시간으로 반영되지 않는 이슈가 발생했을 때, 백엔드 웹소켓 발행 주소와 정합성 로그를 기반으로 역추적하여 데이터가 정상 방출됨을 증명. 
    * 이후 프론트엔드 단에서 "알림이 1개 초과일 때만 화면에 그리도록 한 조건문 설정 오류"를 찾아내어 결함의 위치를 명확히 격리하고 해결을 지원함.

## 6. DBA 역할 연속 수행 및 DDL 선제 최적화

팀원들의 소통 지연과 인프라 파편화 상황 속에서, 데이터베이스 설계 리더십을 발휘하여 개발 병목을 돌파했습니다.

* **파편화된 ERD 단일화 및 통합 제어**: 모듈 4 재정비 중 팀원이 요구사항을 임의로 반영해 ERD 버전을 무분별하게 새로 파편화시키는 리스크 포착. 슬랙 논의의 한계를 인지하고 직접 구두 소통을 주도하여 모든 요구사항을 통합한 모듈 4 최종본 ERD로 단일화 완료 후 파편화된 구버전은 과감히 폐기 처리.
* **자발적 DDL 수립**: 팀원들의 무분별한 수정에도 ERD와 하나하나 비교하여 DDL 구문 정합성을 맞추기 위해 노력함.

## 💡 구현 특징 요약

1. **다대다 패러다임 전환 완수**: 모듈 3의 1:1 구조적 한계를 넘어 다대다 다중 참여 기능과 테이블 확장을 완수했으며, 친구 해제 및 차단 등 실시간 유저 상태 변화를 메시지 발신 타임라인에 결합 가공하는 정밀 로직을 구축함.
2. **철저한 예외 케이스 방어**: 일괄 읽기 시 권한 검증, 중복 자원 제외, 그리고 상호 차단 유저까지 검증하는 신규 방명록 예외 레이어까지 도입하여 웹소켓 및 커뮤니티 환경의 레이스 컨디션과 데이터 정합성 오류를 완벽 차단.
3. **멀티 채널 웹소켓 및 구조적 JSON 설계**: 화면 UI 목적에 따라 구독 채널을 정밀 분리하고, 프론트엔드 조작 효율성을 위해 데이터가 비어있어도 구조를 유지하는 100% 정합성 JSON 포맷 정립.
4. **주도적인 DDL 생성**: 팀 내 ERD 파편화 문제를 소통과 과감한 단일화로 해결하고 ERD 수정 사항을 일일이 비교하며 소통 부재에도 주도적으로 DDL 수립.
