# 📔 모듈 4: 웹 성능 최적화 및 모니터링 시스템 실무 기록
> 애플리케이션의 안정적인 서비스 운영을 위해 Spring Boot Actuator 기반의 지표 수집부터 Prometheus와 Grafana를 활용한 시각화, Loki 및 Promtail 중심의 분산 로그 관리 시스템을 구축했습니다. 아울러 k6 기반의 단계별 부하 테스트를 설계하여 병목 구간을 탐지하고, 근거 중심의 아키텍처 최적화(인덱싱, 캐싱, N+1 개선)를 실현한 실무 학습 기록입니다.  

---

## 📊 애플리케이션 모니터링 및 메트릭 표준화
### 1. Spring Boot Actuator 기반의 시스템 감시 체계

* **애플리케이션 전반의 관찰력(Observability) 확보**: 자바 jvm 메모리, 데이터베이스 커넥션 풀(Connection Pool), CPU 사용량 등 핵심 시스템 내부 리소스를 추적하는 Spring Boot 내장 감시자 역할을 수행합니다.  

* **핵심 엔드포인트(Endpoint) 활용**: `/actuator`, `/health`, `/info`, `/metrics` 및 프로메테우스 연동을 위한 `/actuator/prometheus` 엔드포인트를 노출하여 시스템의 종합적인 기동 상태와 성능 지표를 외부 툴로 내보냅니다.  

* **근거 기반의 코드 평가**: 주관적인 판단이나 눈으로만 코드를 평가하는 것에서 벗어나, 시스템 지표를 바탕으로 성능 차이를 정량적으로 검증하고 수정이 필요한 병목 구간을 객체화합니다.  

### 2. Micrometer 기반 커스텀 메트릭(Custom Metric) 설계

* **메트릭(Metric)과 로그(Log)의 역할 분리**: 메트릭은 수치적 집계(집계된 숫자)를 기반으로 시스템의 이상 징후를 빠르게 발견하는 데 활용하고, 로그는 특정 요청의 흐름을 추적하여 문제의 명확한 원인을 파악하는 구조로 이원화합니다.  

* **핵심 Meter 컴포넌트의 실무 적용**:  

    * **Counter (누적 측정)**: 주문 성공 횟수, 인기 상품 API 호출 누적 횟수와 같이 감소하지 않고 지속해서 증가하는 단방향 이벤트 개수를 수집합니다.  

    * **Timer (지연 및 횟수 기록)**: 실행 시간과 호출 수를 동시에 추적하며, 비즈니스 코드 내부에서 `Timer.Sample`과 `try-finally` 블록을 조합하여 예외 발생 여부와 상관없이 온전한 비즈니스 소요 시간을 기록합니다.  

    * **DistributionSummary (분포 관찰)**: 시간이 아닌 값의 크기를 측정하며, 주문당 상품 종류 수나 주문 금액 분포 등 비즈니스 상태를 입체적으로 대변하는 지표를 수집합니다.

## 📈 분산 시각화 인프라 구축 (Prometheus & Grafana)
### 1. Prometheus 기반 시계열 데이터 관리

* **풀링(Pulling) 메커니즘 연동**: 표준화된 Micrometer 표현식(점 `.` 구분)을 프로메테우스용 수집 규격(언더바 `_` 구분)으로 자동 변환 연동해 주는 구현체 의존성을 주입합니다.  

* **스크랩(Scrape) 정책 수립**: `prometheus.yml` 설정 파일 내부에 수집 주기(`scrape_interval: 5s`)와 타겟 서버 주소를 정의하여, 5초 주기로 애플리케이션의 엔드포인트를 호출하고 시계열 DB에 메트릭을 동적으로 적재합니다.  

* **PromQL 활용**: 프로메테우스 전용 질의어인 PromQL을 기반으로 분 단위 인입 트래픽 및 URI별 지연 시간 흐름을 모니터링합니다.  

### 2. Grafana를 이용한 통합 대시보드 시각화

* **모니터링 아키텍처 연계**: `App[Spring] -> Prometheus -> Grafana` 파이프라인 구조를 정립했습니다.  

* **운영 지표 통합 시각화**: 데이터 소스로 프로메테우스를 연결하고 전용 대시보드 템플릿을 연동하여 평균/최대 응답 시간(ms), 가동 중인 SQL 쿼리 수, SQL 수행 시간 등을 가시화된 차트로 제어합니다.

## 🪵 중앙 집중식 분산 로그 수집 아키텍처 (Loki & Promtail)
### 1. Logback 커스터마이징 및 실무 로그 관리

* **수집 인프라 분리**: 가독성이 떨어지는 인텔리제이 콘솔 출력에 의존하지 않고, 애플리케이션 내부의 부하를 줄이기 위해 외부 파일 저장 및 중앙 집중식 전송 구조를 택합니다.  

* **logback-spring.xml 설정 최적화**: 콘솔 출력 형식을 표준화하는 동시에 별도 파일 시스템(`./logs/`)에 저장하며, 날짜별로 누적 및 분리 저장하고 보관 기간(7일)이 지나면 자동으로 폐기되도록 생명주기를 설정합니다.  

* **MDC 및 TraceId 추적**: `OncePerRequestFilter`와 연계하여 하나의 HTTP 요청 스레드가 수행되는 전 과정 동안 동일한 `traceId`를 유지하도록 바인딩함으로써 로그의 추적 무결성을 완성합니다.  

### 2. Promtail - Loki - Grafana 실시간 연계 파이프라인

* **실시간 로그 스크랩 및 변환**: 로컬 인프라 및 추후 Docker 가상화 환경에 대응하도록 6개 이상의 서버 인프라(Application, Prometheus, Grafana, Loki, Promtail, MySQL, Redis) 유기적 작동 순서를 준수합니다.  

* **Promtail 에이전트 설정**: `promtail.yml` 내에 로그 긁어오기 대상 경로(`__path__: /logs/*.log`)를 지정하고, 포트 `3100`으로 기동 중인 Loki 서버로 전달하는 클라이언트 url 설정을 확보합니다.  

* **LogQL 기반의 정밀 탐색**: Grafana 디렉토리 내에서 LogQL을 활용하여 불필요한 프로메테우스 5초 주기 스크랩 로그를 제거하거나, 유효성 검증 예외 에러 등 특정 장애 조건을 유연하게 모니터링합니다.  

    * **익스프레션 예시**: `{job="popular-shop"} |= "event=request_completed" != "uri=/actuator/prometheus"`

## 🧪 k6 기반 단계별 부하 테스트 (Performance Testing)
### 1. 시나리오 기반 가상 유저(VU) 모델링

* **Smoke Test (00-smoke-check.js)**: 성능 한계를 측정하기 전 가상 유저 1명(`vus: 1`)과 고정 횟수(`iterations`) 반복을 통해 전체 API 파이프라인이 명백한 에러 없이 정상 응답 속도를 내는지 무결성을 확인합니다.  

* **Baseline Load Test (01-popular-baseline.js)**: 단계적 사용자 증가(`ramping-vus`) 및 `stages` 제어 방식을 도입하여, 30초간 5명 증가 $\rightarrow$ 1분간 10명 유지 $\rightarrow$ 30초간 감소 구간을 설계하고 트래픽 가중에 따른 응답 시간 추이를 추적합니다.  

* **Mixed Traffic Test (02-mixed-shop-traffic.js)**: 실제 유저의 행동 패턴과 유사하게 조회 중심 트래픽 80%와 생성/수정(Write 부하) 트래픽 20%를 혼합 인입시켜 동시성 경쟁 상태 및 인프라 한계를 모니터링합니다.  

* **Constant Arrival Rate Test (03-popular-arrival-rate.js)**: 가상 유저 수의 수치에 종속되지 않고, 초당 발생시킬 목표 요청 수인 목표 RPS(`arrival-rate`)를 고정한 채 서버가 특정 처리율 기준을 온전히 버텨내는지 성능 목표를 평가합니다.  

### 2. 성능 지표 평가 및 임계치(Threshold) 검증

* **Tail Latency (상위 1% 극단적 지연) 관리**: k6 기본 통계(p90/p95) 외에 극단적으로 느린 tail latency를 설명할 수 있도록 p99 통계 지표(`summaryTrendStats`)를 명시적으로 세팅하여 분석 가치를 극대화합니다.  

* **비즈니스 예외와 HTTP 실패 구별**: 검증 실패(400 Bad Request)나 재고 부족 예외 발생 등 의도된 비즈니스 실패 케이스들을 무조건적인 HTTP 통신 실패(`http_req_failed`)로 분류하지 않고, `check` 기능을 활용해 기대한 에러 처리가 안정적으로 수행되고 있는지 판정합니다.

## 🏗️ 지표 중심의 아키텍처 최적화 및 유지보수 전략
### 1. 모니터링 결과 기반 성능 개선 가이드 (Metric Target 지점)

* **친구 및 소셜 도메인의 인프라 부하 요인 도출**: 친구 검색, 친구 요청, 채팅방 조회 및 개설, 메시지 내역 조회 등 트래픽이 집중되는 핵심 비즈니스 로직에 호출 횟수, 응답 지연 시간, 에러율 메트릭을 심어 의사결정의 근거를 수집합니다.  

* **성능 병목 요인별 해결 방안 수립**:  

    * **풀 스캔(Full Scan) 부하 제어**: 메시지 검색 시 전체 내역 조회를 방지하기 위해 방 번호와 사용자 아이디 결합 중심의 인덱스(Index) 최적화를 구성합니다.  

    * **조회 지연 및 고부하 CPU 해결**: 검색어 포함 사용자 조회 시 유저 테이블 전체 탐색으로 지연 시간이 길어질 경우, Redis 캐싱 기술을 앞단에 배치하여 데이터베이스 인프라 조회를 차단합니다.  

    * **데이터베이스 N+1 문제 원천 차단**: 메시지 내역 조회 시 연관 엔티티 로딩 과정에서 발생하는 N+1 쿼리 폭탄을 제거하기 위해 JPQL Fetch Join 또는 레포지토리 레이어 상단에 `@EntityGraph`를 도입해 단일 쿼리로 묶어 처리합니다.  

    * **대량 쓰기(Write) 최적화**: 메시지 읽음 처리 시 건건이 나가는 더티 체킹/수정 쿼리 부하를 줄이기 위해 bulk 연산용 `@Modifying` 벌크 쿼리를 선언하여 단 한 번의 업데이트 명령으로 상태를 일괄 동기화합니다.

## 🛠️ [부록] 코드 리뷰 규칙 설계 (주석 기반 가치 내재화)

> 비중이 낮은 주석 필터 지침에 근거하여 아키텍처 설계 사상을 기록합니다.  

* **클린 아키텍처(Clean Architecture) 레이어 가이드라인 수립**:  

    * **Controller**: 비즈니스 로직, 쿼리 조립 및 메트릭 직접 계산을 전면 금지하고, `@Valid` 검증 방어선 구축 및 공통 응답 포맷 준수 여부만 검증합니다.  

    * **Application / Service**: 유스케이스 단위의 도메인 흐름 조율에 충실하며, 단순 지표 수집(Metric) 로직 실패가 핵심 비즈니스 트랜잭션 롤백으로 전파되지 않도록 트랜잭션 경계를 방어합니다.  

    * **Domain**: 기술 인프라(Spring, JPA 등)에 무분별하게 의존하지 않도록 분리하고, public setter 사용을 지양하는 객체 캡슐화 원칙(Tell, Don't Ask)과 도메인 이벤트 발행 관점을 유지합니다.  

    * **Repository / Query**: 대용량 데이터 환경에서 성능 문제를 일으키는 Offset 페이징 구조 대신 Keyset 페이징 전환을 검토하고, 인덱스를 무력화하는 Like문 검색 처리 및 불필요한 엔티티 로딩을 방지하기 위한 DTO 프로젝션(Projection) 설계를 지향합니다.
