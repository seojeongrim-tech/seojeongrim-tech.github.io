## 1. 개요
이번 프로젝트에서 저는 학생 서비스의 핵심인 수강신청 로직과 사용자 간 커뮤니케이션(메시지) 시스템을 담당했습니다. 단순한 기능 구현을 넘어, 실제 학사 행정에서 발생할 수 있는 예외 상황들을 데이터베이스와 비즈니스 로직 수준에서 해결하는 데 집중했습니다.

---

## 2. 해결 과정 및 설계 의도 (Solution & Design)

### 2-1. 수강신청 제약 로직의 고도화
단순히 데이터를 넣는 것을 넘어, 여러 테이블을 참조하여 신청 가능 여부를 판단하는 복합 로직을 구현했습니다.

* **동적 학점 제한**: 학생의 전체 평점을 계산하여 수강 가능 학점을 차등 부여했습니다. (4.0 이상: 25학점 / 2.0 미만: 10학점 등)
* **유효성 검증 순서 최적화**: 서버 자원 낭비를 줄이고 사용자에게 빠른 피드백을 주기 위해 검증 순서를 전략적으로 배치했습니다.
    * **동작 순서**: 중복 과목 체크 → 평점 기반 학점 제한 → 수강 인원 체크 → 시간표 중복 체크

### 2-2. DTO 구조 개선을 통한 협업 효율화
팀 프로젝트 중 공통 DTO를 수정하다 발생하는 코드 충돌 문제를 해결하기 위해 조회 방식을 개선했습니다.

* **[Problem]**: 테이블에 없는 컬럼(교수명, 과목명 등)을 보여주기 위해 DTO에 변수를 계속 추가하다 보니 팀원 간 코드 충돌이 잦아짐.
* **[Solution]**: DTO는 테이블 구조(Entity)를 최대한 유지하되, SQL Join을 활용해 필요한 데이터를 한 번에 가져왔습니다. 부족한 데이터는 Setter를 커스텀하거나 쿼리문에서 해결하여 DTO의 순수성을 지키고 협업 생산성을 높였습니다.

### 2-3. 메시지 시스템의 데이터베이스화(9개 팀 중 유일)
기존의 휘발성 메시지 기능을 DB 기반으로 전환하여 소통의 연속성을 확보했습니다.

* **채팅형 UI 구현**: 발신일과 발신자 정보를 저장하여 대화 흐름을 한눈에 볼 수 있도록 설계했습니다.
* **안 읽은 메시지 계산**: 발신자 ≠ 로그인 사용자이면서 해당 채팅방의 마지막 메시지를 확인하지 않은 경우를 쿼리로 계산하여 알림 기능을 구현했습니다.

## 3. 구조 및 데이터 흐름 (Architecture & Data Flow)

### 3-1. 수강신청 로직 Flow
사용자가 신청 버튼을 누른 순간부터 DB 저장까지의 엄격한 검증 단계입니다.

1.  **Request**: Student ID, Class No 전달
2.  **Validation Step**:
    * **Enrollment**: 이미 신청한 과목인가?
    * **Grade**: 현재 평점으로 이 과목(학점)을 신청할 수 있는가?
    * **Class**: 강의실 정원이 초과되지 않았는가?
    * **Schedule**: 기존 수강 과목과 시간이 겹치지 않는가?
3.  **Response**: 최종 승인 또는 실패 사유(안내 문구) 반환

## 4. 트러블슈팅 (Troubleshooting)

> **❌ 단순 실수로 인한 데이터 누락 (Null 이슈)**
> * **문제**: 기능 테스트 중 학생 이름, 과목명 등이 계속 null로 출력됨.
> * **원인**: DTO에 변수를 추가할 때 IDE 단축키(Alt+Enter)만 사용하여 기본 생성자만 만들고, 정작 데이터를 담아줄 Getter/Setter를 누락함.
> * **해결**: 필요한 필드에 대한 접근자/설정자를 정확히 생성하여 데이터 바인딩 문제 해결.

> **❌ 예외 처리 미흡으로 인한 프로그램 종료**
> * **문제**: 없는 과목을 신청하거나 중복 신청 시 Exception이 발생하며 프로그램 자체가 꺼짐.
> * **원인**: 서비스 레이어에서 예외를 던지기만(throw) 하고 적절히 수습하지 않음.
> * **해결**: try-catch 대신 boolean 반환이나 안내 문구 리턴 방식을 활용하여, 에러 발생 시에도 사용자가 다른 작업을 계속할 수 있도록 가용성 확보.

## 5. 결과 및 성과 (Results)
* **안정성**: 예외 상황에서도 프로그램이 종료되지 않는 견고한 로직 완성.
* **UX 개선**: 테이블 조인을 통해 사용자에게 필요한 정보(교수명, 현재 신청 인원/제한 인원)를 직관적으로 제공.
* **주도적 제안**: 회의 중 '수강 제한 인원' 컬럼 추가를 건의하여 실제 학사 시스템과 유사한 수준으로 데이터 모델 고도화.
* **시연 방식 개선**: 강사-학생의 실시간 상호작용을 직관적으로 보여주기 위해 콘솔 창을 이분할하는 방식을 제안, 9개 팀 중 유일하게 적용하여 발표 효과를 극대화

## 6. 다음 개선 단계 (Next Steps)
* **동시성 제어**: 수강신청 버튼을 동시에 누를 때 발생하는 Race Condition 해결을 위한 DB Lock 메커니즘 학습.
* **기능 확장**: 현재의 쪽지 수준을 넘어 실시간성을 강화하고, '수강바구니(예비수강신청)' 시스템 도입 검토.
* **LMS 본연의 기능**: 강의 등록, 학습 시간 추적(진도율) 등 실제 교육 관리 기능을 추가하여 시스템 완성도 향상.
* * **메시지 조회 방식 개선**: 현재는 채팅방 목록 조회 시 반복문 내에서 각 메시지 건을 개별 쿼리로 조회하는 방식으로 구현되어 있는 구조적 한계가 있다.
