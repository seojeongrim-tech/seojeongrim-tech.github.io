# 📔 모듈 5: AI 연동, 보안/데이터 통합 및 실시간 인프라 고도화 회고

---

## 1. 개요 및 핵심 요약

### 🎯 모듈 5 개요
모듈 5에서는 **메시지 자동 암호화/복호화, Flyway 기반 DB 마이그레이션 관리, 대규모 배포용 더미 데이터 정합성 검증, 웹소켓 실시간 이벤트(타이핑 중 안내 문구 등)** 등 시스템 전반의 보안·데이터·실시간성을 완결짓는 고도화 작업을 진행했습니다.

### 📜 핵심 성과 및 개선 요약
* **데이터 보안 및 안전성 강화**: `AttributeConverter` 기반 **메시지 자동 암호화/복호화** 메커니즘을 적용하고, 암호화 키를 인텔리제이 환경변수 및 `.env`로 격리 관리하여 DB 누출 시 발생할 수 있는 보안 리스크를 사전 차단했습니다.
* **DBA 리더십 및 형상 관리**: 팀 내 **Flyway 도입 후 발생한 네이밍/Use 구문 충돌 및 DDL 파편화 이슈를 주도적으로 해결**하고, 수백 건에 달하는 대규모 배포용 더미 데이터의 연관관계(가입일 전 활동 기록, 수강평 실제화, 완료 여부 등) 정합성을 수송·검증했습니다.
* **사용자 경험(UX) 및 실시간성 극대화**: **메시지 입력 중(Typing Indicator) 실시간 웹소켓 이벤트** 구축, 클라이언트 테스트용 HTML 스크립트 작성 및 CORS/Security 환경 구성을 통해 프론트엔드 연동 신뢰성을 높였습니다.
* **클라이언트 맞춤형 API 스펙 리팩토링**: 도시별 방명록 목록 조회 쿼리 스트링 전환, 채팅방 멤버 목록 최상단 "(나)" 고정 및 정렬 가공, 친구 요청 수락/거절 시 알림 자동 읽음 처리 등 정밀한 UX 개선을 이끌었습니다.

## 2. 해결 과정 및 설계 의도 (Solution & Design)

### 2-1. 메시지 엔티티 자동 암호화/복호화 (Security & Privacy)
[클라이언트/서비스] ──(원문 텍스트)──► [MessageJpaEntity] ──(AttributeConverter)──► [DB: AES 암호문]
[클라이언트/서비스] ◄──(원문 텍스트)─── [MessageJpaEntity] ◄──(AttributeConverter)─── [DB: AES 암호문]

* **AS-IS**: 메시지 본문(`content`)이 DB에 평문으로 저장되어, DB 유출 시 개인정보 및 대화 내역이 노출될 위험이 존재했습니다.
* **TO-BE**: `AttributeConverter` 기반의 메시지 암호화 클래스를 구현하고 `MessageJpaEntity`에 어노테이션을 적용하여, **DB 저장 시 자동 암호화, 조회 시 자동 복호화**가 이루어지도록 처리했습니다.
  * 기존 평문 더미 데이터와의 호환성을 고려하여, 복호화 실패 시(암호화되지 않은 기존 데이터) 원문 형태를 그대로 반환하도록 예외 안전망을 구축했습니다.
  * 암호화 키는 yml에 직접 노출하지 않고, `.env` 및 인텔리제이 환경변수로 분리하여 중앙 관리하고 유실 시 복호화 불가 문제를 인지하여 별로 보관함으로써 보안 사고를 예방했습니다.

### 2-2. 메시지 입력 중(Typing) 웹소켓 인프라 및 가공 로직

* **요구사항**: 사용자가 채팅방에서 메시지를 입력 중일 때, 해당 방에 속한 타 사용자들에게 실시간으로 입력 상태를 방출해야 함.
* **구독 및 발행 구조**:
  * **발행 (Pub)**: `/pub/chat/typing/{roomId}` (`{ isTyping: true/false }`)
  * **구독 (Sub)**: `/user/sub/chat/typing/{roomId}`
* **인메모리 문구 가공 파이프라인**:
  * 입력 중인 인원수에 따라 가공 문구를 동적으로 생성하여 반환합니다.
    * 0명: `""`
    * 1명: `"민수님이 입력중입니다."`
    * 2명: `"민수님, 승준님이 입력중입니다."`
    * 3명 이상: `"민수님 외 2명이 입력중입니다."`
      * (항상 같은 사람을 위치시키기 위해 본인 제외한 가나다 순으로 정렬 후 처리)
  * 테스트 편의성을 위해 포스트맨의 한계를 극복하는 **독립 HTML/JS 웹소켓 테스트 스크립트**를 작성하여 F12 콘솔을 통해 사전 검증을 완결했습니다.

### 2-3. Flyway 기반 DB 마이그레이션 통제 및 대규모 더미 데이터 수선

* **Flyway 마이그레이션 규칙 정립**:
  * 다른 DBA가 도입한 Flyway 적용 과정에서 발생한 Flyway 규칙 미숙으로 인한 파일명 컨벤션 미준수(`__` 두 개 누락)를 해결 및 학습했고 스크립트 내부 `USE [DB_NAME]` 구문 삽입으로 인한 타 로컬 DB 실행 에러를 발견했습니다.
  * SQL 스크립트에서 `USE` 구문을 일괄 제거하고 Flyway 파일명 규칙을 바로잡아 팀 공통 브랜치 빌드 깨짐 현상을 수습했습니다.
* **대규모(Large) 배포 데이터 정합성 수선**:
  * 사용자 가입일(`created_at`), 수강 상태(`enrollment`), 진도율(`total_progress`), 수강 완료 여부(`is_completed`) 간의 불일치로 인해 AI 요약 및 기능 테스트가 차단되는 잠재적 문제를 파악했습니다.
  * 타 도메인의 더미 데이터 연관관계를 직접 전수 검증 및 수선하여, 배포 환경에서 발생할 수 있는 유효성 에러를 사전에 방지했습니다.

## 3. 트러블슈팅 (Troubleshooting)

### 3-1. Flyway 마이그레이션 및 DBA 형상 관리 이슈

#### ❌ 매 실행 시 Flyway 스키마 히스토리 충돌 문제
* **현상**: 프로젝트 실행마다 Flyway 스키마 히스토리가 기존 DB 테이블과 충돌하여 애플리케이션 구동 실패[cite: 2].
* **원인 및 해결**: 로컬 DB 환경에서 공용 DDL 구조가 변경될 때 스키마 이력이 꼬이는 현상으로, 매 실행 시 `DROP TABLE flyway_schema_history;` 구문을 실행하여 히스토리를 초기화하고 공용 DDL에 `DROP` 구문을 주기적으로 보완하여 해결했습니다[cite: 2].
* **인사이트**: AI(CodeRabbit)가 파일 분리를 제안했으나, 개인 로컬 DB 환경의 특성을 고려해 공용 DDL 통합 방식을 고수했습니다[cite: 2]. 팀원들이 소통 없이 ERD를 변경할 때도 프로젝트를 직접 풀 받아 일일이 대조·수정해가며 DBA로서 스키마 정합성을 수호했습니다[cite: 2].

#### ❌ Flyway 파일명 컨벤션 및 USE 구문 에러
* **현상**: 팀원이 도입한 Flyway 마이그레이션 적용 시 타 팀원 로컬 환경에서 지속적 빌드 에러 발생[cite: 2].
* **원인**: SQL 파일 내 특정 DB를 지정하는 `USE` 구문 남발 및 파일명 명명 규칙(`V1__` 등) 미준수[cite: 2].
* **해결**: 전사 SQL 파일의 `USE` 구문을 일괄 삭제하여 PR을 올렸고, 언더바 2개(`__`) 규칙을 적용하여 정상 실행시켰습니다[cite: 2]. 이후 팀원들에게 파일 명명 규칙과 더미 데이터 추가 위치(`common` 하단)를 가이드했습니다[cite: 2].

### 3-1. 프론트엔드 연동 및 API 스펙 정밀 조정

#### ❌ 방명록 조회 시 도시 기준 식별 불가 및 URL 매핑 결함
* **현상**: 타인의 도시 방문 시 내 방명록 목록 조회가 정상 동작하지 않음.
* **원인**: URL PathVariable로 전달받던 방식에서는 프론트엔드가 해당 도시가 로그인 유저의 도시인지 판단할 수 없는 구조적 한계가 존재했습니다.
* **해결**: Request Parameter(쿼리 스트링) 방식으로 전환했습니다. 쿼리 스트링 값이 넘어오면 타 도시 조회의 맥락으로 판단하고, 값이 없으면 컨트롤러/서비스 레이어에서 로그인 유저의 ID를 자동 주입하도록 처리했습니다. (프론트의 늦은 확인으로 Id가 본인인 경우, 아닌 경우로 서비스 레이어에서 처리했기에 재작업에선 컨트롤러만 수정하여 개발 속도 향상시킴)

#### ❌ 채팅방 멤버 목록 조회 시 정렬 및 표시 규격 미흡
* **현상**: 멤버 목록 조회 시 본인 위치 파악이 어렵고 가독성이 떨어짐.
* **해결**:
  1. 접근 권한(방 참여 여부)을 먼저 검증하여 미참여 시 예외 메시지 반환.
  2. 조회 결과 중 **로그인 유저는 최상단에 고정**하고 닉네임 뒤에 `"(나)"` 수식어 부여.
  3. 나머지 멤버는 가나다 순으로 정렬하며, 역할(강사/학생)에 따른 이름(`name`) 필드 노출 제어를 적용했습니다.

### 3-2. 보안, 환경 변수 및 Git 형상 관리 트러블 슈팅

#### ❌ `.env.example`을 통한 API 키 유출 위험 예방(부트캠프 강의 수업 관련)
* **현상**: 외부 라이브러리 및 소스 푸시 과정에서 Git 보안 경고 발생.
* **원인**: `.gitignore`에 `.env`만 등록되어 있고 `.env.example` 파일에 실제 API Key 예시가 포함되어 유출될 위험이 존재했습니다.
* **해결**: `.env.example` 내의 하드코딩된 API Key를 제거하고 단순 키 이름 형태로 정제하였으며, 환경 변수를 통해서만 설정 값이 주입되도록 보완했습니다.

#### ❌ 프론 포트(4444) CORS 및 Security 허용 결함
* **현상**: 프론트엔드 배포/테스트 환경에서 웹소켓 연결 시 CORS 에러 발생.
* **원인**: `SecurityConfig` 및 `WebSocketConfig` 내에 해당 테스트 포트(4444)에 대한 허용 정책 및 중복 역할 코드가 존재했음[cite: 2].
* **해결**: 중복된 웹소켓 설정 코드를 정돈하고 `SecurityConfig` 내에 포트 및 출처 허용 정책을 바르게 재구성하여 연동을 완결했습니다[cite: 2].

### 3-3. 팀 협업 및 도메인 경계 관리 트러블 슈팅

#### ❌ 타 팀원의 도메인 영역 무단 수정 및 침범
* **현상**: 병합 과정에서 타 팀원이 알림/메시지 도메인 내부 코드나 포트를 임의로 수정하거나 주석을 해제하여 충돌 발생[cite: 2].
* **해결**: 충돌 발생 시 내 도메인의 원본 비즈니스 로직을 우선적으로 보존하는 방향으로 PR을 재정리하고 머지를 진행했습니다[cite: 2]. 도메인 간 이벤트 연동 시 직접 클래스 참조 대신 이벤트 발행/구독(Listener) 구조를 준수하도록 가이드를 공유했습니다.

#### ❌ 외부 제공 코드 복사 시 VSCode 특수 공백문자 인코딩 에러(부트캠프 강의 수업 관련)
* **현상**: 강사 제공 소스 코드를 복사하여 붙여넣을 때 연속적인 컴파일 에러 발생[cite: 2].
* **원인**: 들여쓰기가 비표준 공백문자(`\u00A0`)로 인코딩되어 띄어쓰기로 잘못 인식됨[cite: 2].
* **해결**: AI 제안을 통해 VSCode 단축키(`Ctrl + H`)를 활용해 `\u00A0`를 일반 공백(스페이스)으로 일괄 치환해 에러를 즉시 해결했습니다[cite: 2].

## 5. 한계 및 향후 개선 과제 (Limitations & Next Steps)

* **JPA 엔티티 간 강결합 문제**:
  * 빠른 개발 과정에서 타 도메인과의 연관관계 처리를 위해 `JpaEntity`를 직접 참조하는 구조가 일부 남아있습니다[cite: 3]. 향후 완전히 독립된 바운디드 컨텍스트(Bounded Context) 구축을 위해 식별자(`Id`) 기반 참조 및 비동기 이벤트 통신 구조로 보완이 필요합니다.
* **ENUM 활용을 통한 타입 안전성 확보**:
  * 알림 타입 및 메시지 상태 값 처리 시 ENUM 클래스를 작성해 두었으나, 일부 로직에서 문자열(`String`) 형태로 직접 비교/처리하는 부분이 존재합니다[cite: 3]. 도메인 안정성을 위해 향후 ENUM 기반의 타입 안전 코드로 리팩토링할 예정입니다[cite: 3].
